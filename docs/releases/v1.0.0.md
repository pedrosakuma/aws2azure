## Install exact promoted bytes

### Container

Use the digest, not a mutable convenience tag:

```bash
docker pull ghcr.io/pedrosakuma/aws2azure@sha256:81e1152be4028de1b380007d07b7596cf12ecb032c1f93e51db5a8d26d2bf9bf
docker run --rm \
  -p 8080:8080 \
  -v "$PWD/config.json:/app/config.json:ro" \
  -e AWS2AZURE_CONFIG_FILE=/app/config.json \
  ghcr.io/pedrosakuma/aws2azure@sha256:81e1152be4028de1b380007d07b7596cf12ecb032c1f93e51db5a8d26d2bf9bf
```

The `v1.0.0` GHCR tag is an alias for this exact existing RC index; promotion
does not build or push a different image.

### Native AOT archive

Download the archive for the target architecture from the `v1.0.0` GitHub
Release, then verify it before extraction:

```bash
sha256sum --check SHA256SUMS.all
tar -xzf aws2azure-v1.0.0-rc.1-linux-x64-e157662204a51d94748197c01140e3c2eb8aa09fbb763adab5fa19e56a0afa55.tar.gz
```

Use the `linux-arm64` archive and its checksum on arm64 hosts.

## Upgrade and rollback

This is the first stable `v1` release. Configuration schema 1 and the persisted
formats listed in the
[compatibility policy](../versioning-and-compatibility.md) are unchanged by the
RC-to-stable promotion.

Before rollout, retain the exact prior artifact and configuration as required by
the [production runbook](../deployment/production-runbook.md). The qualified
pre-v1 rollback baseline used by both observations is:

- source `fd72cda28ca68ad53cd6b30fb3f43cbe923580ef`;
- sealed artifact `8418646730`, run
  [29610742551](https://github.com/pedrosakuma/aws2azure/actions/runs/29610742551);
- aggregate digest
  `sha256:6d4476c6bcce82fb3d47f72f007a2d81d95f504fc95f9671a0821d8cd513bbef`.

Repository collaborators can retain it before rollout with:

```bash
gh run download 29610742551 \
  --repo pedrosakuma/aws2azure \
  --name aws2azure-sealed-linux-x64-6d4476c6bcce82fb3d47f72f007a2d81d95f504fc95f9671a0821d8cd513bbef-run-29610742551-attempt-1
```

To roll back, stop traffic to `v1.0.0`, restore that retained executable and the
matching previous configuration, verify `/health`, `/ready`, and the module
list, then restore traffic. Rollback restores code and configuration; it does
not undo Azure writes or credential changes. Both qualification profiles
verified candidate-write/previous-read behavior and exact-prior restoration.

## Known incompatibilities and operator actions

- aws2azure translates AWS wire protocols to selected Azure data planes; it is
  not an AWS IAM, control-plane, or full-service emulator.
- Support is limited to the two GA profiles above. Review every required
  operation and accepted gap before production use.
- S3 clients must use path-style addressing through the proxy.
- Azure topology, capacity, consistency, throttling, and authorization semantics
  remain Azure-native and can differ from AWS.
- The pre-v1 rollback artifact is a retained workflow artifact, not a supported
  stable release. Operators must preserve it before starting the first rollout.
